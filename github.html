Lección 1

El repositorio no es para ti

Uno de los errores más comunes es asumir que la documentación existe porque tú ya entiendes el proyecto. La realidad es exactamente la contraria: la documentación existe para las personas que no estuvieron presentes durante el diseño.

Tu README debe servir para que un profesor, compañero, reclutador, fabricante o incluso tu versión futura pueda comprender el proyecto rápidamente.

Lección 2

El README es la portada

Cuando una persona entra a tu repositorio, normalmente verá el README antes que cualquier otro archivo.

Por eso debería responder inmediatamente:

  • ¿Qué es?
  • ¿Para qué sirve?
  • ¿Qué problema resuelve?
  • ¿Cómo se fabrica?
  • ¿Cómo se prueba?
Lección 3

Estructura ideal de carpetas

Un repositorio profesional está organizado por propósito y no por fechas o versiones improvisadas. 

Repositorio/

├── docs/
├── hardware/
├── firmware/
├── fabrication/
├── simulations/
├── images/
├── README.md
└── LICENSE

Cada carpeta tiene una responsabilidad específica. Esto facilita la navegación y el mantenimiento.

Lección 4

Las imágenes son obligatorias

Los proyectos de hardware son visuales. Un repositorio sin imágenes obliga al lector a imaginar cómo luce el sistema.

Imágenes recomendadas

  • Vista superior PCB
  • Vista inferior PCB
  • Vista 3D
  • Diagrama de bloques
  • Foto ensamblada

Beneficio

Permiten entender el proyecto en menos de un minuto.

Lección 5

Cómo escribir bien

La documentación debe explicar decisiones, no únicamente listar archivos.

No aporta valor

Aporta contexto

El sistema utiliza un ESP32 para transmitir mediciones mediante WiFi hacia un servidor remoto.
